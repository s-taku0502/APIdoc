🤔 APIとは？

API（Application Programming Interface）とは、異なるソフトウェア同士が情報をやり取りするためのルールや仕組みのことです。

身近な例で理解しよう

レストランを想像してください：

  • お客様（あなたのアプリ）：料理を注文したい
  • ウェイター（API）：注文を受けて、キッチンに伝える
  • キッチン（サーバー）：料理を作って、ウェイターに渡す
  • 料理（データ）：最終的にお客様に提供される

APIはこのウェイターの役割を果たします。あなたは料理の作り方を知らなくても、ウェイターに注文すれば料理が運ばれてきます。

💡 なぜAPIが必要なの？

APIの利点

  1. 効率性：一から作らず、既存のサービスを活用できる
  2. 専門性：それぞれの得意分野に集中できる
  3. セキュリティ：直接データベースにアクセスせず、安全にデータを取得
  4. 標準化：統一された方法でデータをやり取り

実際の例：

天気アプリを作るとき、世界中の気象データを自分で収集するのは大変です。でも、気象庁のAPIを使えば、簡単に最新の天気情報を取得できます。

APIとは？ APIの種類

🔄 APIの種類

1. REST API（最も一般的）

  • HTTP通信を使用
  • シンプルで理解しやすい
  • JSON形式でデータをやり取り

2. GraphQL API

  • 必要なデータだけを取得できる
  • 1回のリクエストで複数の情報を取得
  • FacebookやGitHubで使用

3. WebSocket API

  • リアルタイム通信が可能
  • チャットアプリやゲームで使用
  • 双方向通信
なぜAPIが必要なの？ HTTPメソッドの基本

🌐 HTTPメソッドの基本

HTTPメソッドは、サーバーに対して「何をしたいか」を伝える動詞のようなものです。

主要な4つのメソッド

GETデータを取得する

📖 本を読む（情報を見るだけ、変更しない）

GET /api/users/123

ユーザーID 123の情報を取得

POST新しいデータを作成する

✏️ 新しい記事を書く

POST /api/users

新しいユーザーを作成

PUTデータを更新する

📝 既存の記事を編集する

PUT /api/users/123

ユーザーID 123の情報を更新

DELETEデータを削除する

🗑️ 記事を削除する

DELETE /api/users/123

ユーザーID 123を削除

APIの種類 リクエストとレスポンス

📨 リクエストとレスポンス

リクエスト（要求）の構成

GET /api/users/123 HTTP/1.1 Host: example.com Authorization: Bearer your-token-here Content-Type: application/json

レスポンス（応答）の構成

HTTP/1.1 200 OK Content-Type: application/json { "id": 123, "name": "田中太郎", "email": "tanaka@example.com", "created_at": "2025-01-15T10:30:00Z" }
HTTPメソッドの基本 ステータスコード

📊 ステータスコード

ステータスコードは、サーバーからの「返事の種類」を示す3桁の数字です。

よく見るステータスコード

成功系（200番台）

  • 200 OK - 成功！欲しいデータが取得できました
  • 201 Created - 新しいデータが作成されました

クライアントエラー（400番台）

  • 400 Bad Request - リクエストに問題があります
  • 401 Unauthorized - 認証が必要です
  • 404 Not Found - 指定されたリソースが見つかりません

サーバーエラー（500番台）

  • 500 Internal Server Error - サーバー側で問題が発生しました
リクエストとレスポンス 認証について

🔐 認証について

認証は「あなたが誰であるか」を証明し、「何にアクセスできるか」を決める仕組みです。

主な認証方式

1. APIキー

最もシンプルな方式。長い文字列をヘッダーに含めて送信。

Authorization: Bearer sk_test_4eC39HqLyjWDarjtT1zdp7dc

2. JWT（JSON Web Token）

ユーザー情報を含んだトークン。有効期限がある。

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

3. OAuth 2.0

「Googleでログイン」「Twitterでログイン」などで使われる方式。

ステータスコード 実践例

💻 実践例

例1：ユーザー情報を取得

// JavaScript（fetch API）を使用 fetch('https://api.example.com/users/123', { method: 'GET', headers: { 'Authorization': 'Bearer your-token-here', 'Content-Type': 'application/json' } }) .then(response => response.json()) .then(data => { console.log('ユーザー情報:', data); }) .catch(error => { console.error('エラー:', error); });

例2：新しいユーザーを作成

fetch('https://api.example.com/users', { method: 'POST', headers: { 'Authorization': 'Bearer your-token-here', 'Content-Type': 'application/json' }, body: JSON.stringify({ name: '佐藤花子', email: 'sato@example.com' }) }) .then(response => response.json()) .then(data => { console.log('新しいユーザーが作成されました:', data); });

例3：Python（requests）を使用

import requests # ヘッダーに認証情報を含める headers = { 'Authorization': 'Bearer your-token-here', 'Content-Type': 'application/json' } # ユーザー情報を取得 response = requests.get('https://api.example.com/users/123', headers=headers) if response.status_code == 200: user_data = response.json() print(f"ユーザー名: {user_data['name']}") else: print(f"エラーが発生しました: {response.status_code}")

🛠️ 便利なツール

API開発・テスト用ツール

1. Postman

  • 視覚的にAPIをテストできる
  • コレクション機能でAPIをまとめて管理
  • 初心者におすすめ

2. curl（コマンドライン）

# GET リクエスト curl -X GET "https://api.example.com/users/123" \ -H "Authorization: Bearer your-token-here" # POST リクエスト curl -X POST "https://api.example.com/users" \ -H "Content-Type: application/json" \ -H "Authorization: Bearer your-token-here" \ -d '{"name":"佐藤花子","email":"sato@example.com"}'

3. ブラウザの開発者ツール

  • F12キーで開く
  • Networkタブでリクエスト/レスポンスを確認
  • Consoleタブでfetch APIを試す
実践例 ベストプラクティス

✅ ベストプラクティス

開発者向けのコツ

1. エラーハンドリングを忘れずに

fetch('/api/data') .then(response => { if (!response.ok) { throw new Error(`HTTP error! status: ${response.status}`); } return response.json(); }) .then(data => console.log(data)) .catch(error => console.error('エラーが発生しました:', error));

2. APIキーは安全に管理

  • 環境変数に保存する
  • コードに直接書かない
  • 公開リポジトリにコミットしない

3. レート制限を理解する

  • 1分間に何回までリクエストできるかを確認
  • 制限を超えないよう調整
  • 429エラーが出たら少し待つ

4. APIドキュメントを読む習慣をつける

  • パラメータの形式を確認
  • レスポンスの構造を理解
  • エラーコードの意味を把握
